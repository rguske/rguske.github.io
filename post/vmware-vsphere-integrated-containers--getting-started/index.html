VMware vSphere Integrated Containers | Getting Started

The first lines are the toughest when you decide to start your first blog post. So many thoughts running through my head when I decided to do this…“How shall I begin?”…“Is it just another blog?”…and the most frightening…“What if YOU, the reader, don´t like what I´m writing about?”…but well…I´d like to handle these thoughts like Tony Stark did it in one of my absolute favorite movies Iron Man (1), where he told J.A.R.V.I.S “SOMETIMES YOU GOTTA RUN BEFORE YOU CAN WALK.”

These times are so exciting and my friend and well appreciate colleague Alex put it into right words when he says, that it isn´t anymore all about Compute-Virtualization. I´m very keen on everything related to Cloud-Native and all topics around the decoupling of a process from the underlying operating system and much more the infrastructure underneath to run those distributed systems which are the result by the end of the day. For me personally, it becomes so present back in 2016 for the first time and much more when I decided to work for VMware. Did you know that VMware is a huge contributor to Open Source Projects and a Platinum Member at the Cloud Native Computing Foundation - CNCF? You´ll find some of our open source projects here.

But I´ll stop here before I get away from the actual and start geeking-out. You are here to read about what the headline promised you and now let us begin with vSphere Integrated Containers aka VIC. Version 1.4 was already launched by our Cloud-Native Business Unit (CNABU) on the 15th of March this year. You´ll find all necessary information for this release under the following links:

  1. Release Notes
  2. Documentation
  3. VMware Cloud Native Blog

Let´s dig into Part I “Getting started”

IMPORTANT! You´ll find all necessary pieces of information with regards to Licensing as well as Deployment Requirements on the official VIC Github Page.

1st, we have to download the latest bits from MyVMware.com here: VIC Version 1.4.0. After you´ve downloaded the 3,12 GB ova-file, we will start provisioning the VIC Virtual Appliance over the vSphere Web Client onto your vSphere Datacenter, Cluster or ESXi Host. I suppose that you are already familiar with these steps but if not, please go here first.

The next step is to complete the VIC appliance installation through the establishment of the connection to our vCenter Server as well as Platform Service Controller. Here we have to enter the vCenter Server address (FQDN) and the Single Sign-on credentials for a vSphere administrator account. In my case, I´m using an embedded PSC and with that, I can leave the fields for the External PSC Instance empty.

If you´ve entered your credentials correct, you´ll be forwarded to the VIC Getting Started page which you could always open by using the IP-Adress or better using the FQDN over port 9443. In my example: https://vic01:9443/

II. Installation of the VIC Plug-In for the vCenter Server Appliance

In this section, I´d like to give you a walkthrough on how to install the VIC HTML5 plug-in for the vCenter Server Appliance. I´m already using the vCenter Server Appliance aka vCSA in Version 6.7.0. In general, it is important to know, that a minimum version of vCenter Server 6.5.0d is required to make use of the plug-in. But! Also worth to mention is, that the plug-in is not necessary or a requirement to run VIC as well as for the deployment of a Virtual Container Host aka VCH. But it makes the initial deployment of a VCH easier at the first beginning until you are more and more familiar with the vic-machine utility. The first step from the installation of the HTML5 plug-in is to download the vic-machine-bundle from the VIC-Appliance by using curl. To set up the necessary commands from within the vCSA we have to first enable SSH over the Virtual Appliance Management Interface aka VAMI over port 5480. By default SSH is disabled.

https://lab-vcsa67-001:5480

Toggle the switch to Enable SSH Login on the vCSA

But wait! To make sure if after the installation all went right, we should check before and after. And how can we check if there isn´t a vic-plugin before the installation and afterward it gets shown? Correct - over the Managed Object Browser aka MOB.

https://lab-vcsa67-001/mob

You´ll find nothing declared with vic- under content/ ExtensionManager/ extensionList.

Now let’s establish a connection to the vCSA by using a terminal session.

ssh root@192.168.100.72

shell

After the login and we now want to download the vic-bundle tar-file from the VIC Appliance to start the installation of the VIC H5 plug-in. You should just change my IP-address to yours in the following lines.

export VIC_ADDRESS=192.168.100.160
export VIC_BUNDLE=vic_v1.4.0.tar.gz
curl -kL https://${VIC_ADDRESS}:9443/files/${VIC_BUNDLE} -o ${VIC_BUNDLE}

You should see the downloaded tar-file by using the Linux command ls (list)-ltr (sort by change date). The next and final step before the installation of the plug-in can start, is to execute the installation script.

tar -zxf ${VIC_BUNDLE}
cd vic/ui/VCSA

Execute the install.sh script by entering:

./install.sh

…and provide the asked data (vCSA FQDN as well as vCenter Server Administrator Credentials). By the end it should look like this:

Now let´s check again the extensionList over the vCenter Managed Object Manager - MOB. Wolla! Okay, now we are close to make use of the vic-plugin capabilities in our vSphere-WebClient but before we can go ahead, we need to restart the H5-Client as well as the Flex-based vSphere Web Client by setting up the following:

service-control --stop vsphere-ui && service-control --start vsphere-ui
service-control --stop vsphere-client && service-control --start vsphere-client

Okay, now we have to get rid of our tracks by starting the cleaning process through the deletion of the unpacked tar-file.

rm *.tar.gz
rm -R vic

The VIC-Plugin should now be available under Menu/ vSphere Integrated Containers.

Now we have established the Integration into vSphere but what about the Containers? It´s not far away anymore…
